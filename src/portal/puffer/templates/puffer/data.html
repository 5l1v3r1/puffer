Data Archive


Along with our research paper, we are publishing an archive of anonymized data collected on Puffer for the research community to investigate. This page contains data between Jan. 2019 and Jan. 2020; new data, along with the results of ongoing experiments on Puffer, will be posted in the experiment results page.


Data Description

Please download a small sample of fake data first to determine if the Puffer data is indeed what you need. If anything is not clear in the below data description, please don't hesitate to post a question in our Google Group.

At a high level, the Puffer data contains a single JSON file that encodes the experimental settings, and each day of data is stored as time series in four CSV files: video_sent_X.csv, video_acked_X.csv, client_buffer_X.csv, where X represents the start of the day. For example, "2019-11-04T11" means the file contains data collected between 2019-11-04T11:00:00Z and 2019-11-05T11:00:00Z (time zone is UTC by default).

  • expt_DAY.json contains the settings of all the past experiments in one place, dumped on a DAY. The key is expt_id, a unique ID to identify an experiment. The value contains experiment configuration such as congestion control, ABR algorithm and the associated parameters. Note that the field abr_name is simply a meaningful name for a specific combination of abr abr_config.
  • video_sent_X.csv: each line represents a data point collected every time when a Puffer server sends a video chunk to a client. It contains the fields below in order:
    • time: timestamp (milliseconds since Unix epoch) when the chunk is sent
    • session_id: unique ID for the video session
    • expt_id
    • channel: TV channel name
    • video_ts: unique presentation timestamp of the chunk. Every chunk in each channel is always 2.002 seconds long, i.e., 180180 at 90 kHz. Therefore, a continuous playback of a channel would require video_ts to increment by 180180 in each chunk.
    • format: encoding settings of the chunk, represented as "WxH-CRF", namely the resolution and constant rate factor (CRF)
    • size: chunk size in bytes
    • ssim_index: SSIM of the chunk (relative to a canonical version of the chunk). Note that it is unitless and not in dB.
    • cwnd: congestion window size in packets (tcpi_snd_cwnd)
    • in_flight: number of unacknowledged packets in flight (tcpi_unacked - tcpi_sacked - tcpi_lost + tcpi_retrans)
    • min_rtt: minimum RTT in microseconds (tcpi_min_rtt)
    • rtt: smoothed RTT estimate in microseconds (tcpi_rtt)
    • delivery_rate: TCP's estimation of delivery rate in byte/second (tcpi_delivery_rate)
  • video_acked_X.csv: each line represents a data point collected every time when a Puffer server receives a video chunk acknowledgement from a client. It contains fields below in order:
    • time: timestamp (milliseconds since Unix epoch) when the chunk is acknowledged
    • session_id
    • expt_id
    • channel
    • video_ts: presentation timestamp of the chunk, which can be used to find the matching chunk in video_sent_X.csv. It is usually monotonously increasing but will be reset to 0 from time to time (e.g., when Puffer's encoding pipeline is restarted). Although it is not guaranteed to be a universally unique ID, duplicate matches within each day of data are extremely rare in practice.
  • client_buffer_X.csv: each line represents a data point reported by the client, when certain events occur and on a regular interval. It contains fields below in order:
    • time: timestamp (milliseconds since Unix epoch) when the client message is received
    • session_id
    • expt_id
    • channel
    • event: type of event reported by the client:
      • init: when the client switches to a new channel
      • startup: when the client starts playing the channel
      • rebuffer: when the client rebuffers
      • play: when the client resumes playing from rebuffering
      • timer: periodic event reported by the client, designed to fire every 250 ms. However, the interval is not reliable and can often be longer, especially when the client's browser tab becomes inactive.
    • buffer: playback buffer size in seconds
    • cum_rebuf: total time in seconds that the client has spent rebuffering when playing the current channel
